# 데이터 → 정확도 · 자가발전(플라이휠) 설계

> 타깃 heavy user(재회러·짝사랑 진지·위기 회복)에게 **더 정확한 결과**를 주기 위한 데이터 전략.
> 핵심 질문: "어떤 데이터를 넣으면 정확해지고, 스스로 좋아지나(self-improving)?"
>
> ⚠️ **현 구현 상태(2026-06)**: MVP의 진단 엔진은 **규칙 기반·결정적**(`web/src/lib/diagnose/engine.ts`)이며 LLM/AI 호출이 없다. 저장되는 데이터는 `diagnoses` + `profiles`(생년·MBTI·애착) 두 테이블이고, 아래에 설명하는 **결과 피드백·자가발전 플라이휠(outcomes/cases/messages/expert_cases, 모델 재학습)은 전부 계획/미구현**이다. 이 문서는 "앞으로 무엇을 넣어 정확도를 올릴지"를 그리는 **로드맵**이다.
>
> ✅ **방금 구현(2026-06)**: 계층1 입력 중 **생년·MBTI·애착·상대정보 수집이 구현됨**. 단 **MBTI·나이차는 점수에 반영하지 않는 '참고 레이어'**(소통 톤 팁·궁합/나이차 한 줄)이고, 생년은 **청소년(연나이 ≤19) 안전 판정**에만 결정적으로 쓰인다. 애착은 기존대로 점수 신호다. MBTI/궁합/나이차는 **참여·개인화용 '참고'이지 예측의 핵심 무기가 아니다**(솔직한 신뢰도 표기). 진단 엔진은 여전히 규칙기반·결정적.

## 1. 정확도를 올리는 데이터 3계층

### 계층 1 — 더 풍부한 입력 피처
점수에 결정적으로 반영되는 핵심 신호는 여전히 **설문 8~11문항(연락 흐름·반응·애착·맥락)**이다(규칙 엔진). 입력 수집은 아래처럼 일부 구현됨:

**✅ 구현됨 — 영속 프로필 + 진단 입력**
- **생년(출생연도)** — 가입 필수, `profiles` 테이블 + user_metadata 저장([web/supabase/schema.sql](../../web/supabase/schema.sql), [web/src/lib/profile.ts](../../web/src/lib/profile.ts)). 용도: ① **청소년(연나이 ≤19) 안전 판정**에 결정적 사용(이전 '10대 선택' 버킷 폐지, 생년 기반으로 변경) ② 나이차 코멘트(참고). 진단 플로우는 로그인+프로필 생년이 있으면 '내 정보' 단계를 생략.
- **MBTI(선택)** — 나/상대 모두 수집. ★**점수 미반영 = 참고 레이어**. 상대 MBTI→소통 톤 팁, 나×상대 MBTI→궁합 한 줄([web/src/lib/diagnose/personality.ts](../../web/src/lib/diagnose/personality.ts), `Diagnosis.compat`).
- **애착 성향** — 프로필에서 편집·저장 가능(이전엔 읽기전용). 진단 설문의 `selfAttach` 신호로 **점수에 반영**(참고 레이어 아님).
- **상대 정보(선택)** — 상대 생년·MBTI를 진단 플로우 '상대 정보' 단계에서 수집(건너뛰기 가능). 제3자 정보라 최소 수집. MBTI→소통 톤, 나이차→코멘트로만 활용(**점수 미반영**).

**🔜 다음 단계 (계획/미구현)** — 추가하면 신호가 더 정밀해짐:
- **대화 캡처/텍스트** (계획/미구현 — 프로토타입에만 있던 메시지 해석, 현 MVP엔 없음) — 상대 온도, **응답 속도 추세**, 누가 먼저, 메시지 길이 변화. 재회·썸에서 가장 강한 실측 신호.
- **연락 타임라인 로그** — 시간에 따른 연락/반응 추세(없음→가끔 등). 스냅샷보다 추세가 예측력 높음.
- **자유서술 활용** (현재 미반영 — 입력은 받지만 점수엔 미반영) — 키워드·맥락 추출로 요인 보정.
- **더 정밀한 애착 진단** — 1문항이 아닌 짧은 척도.

> ⚠️ **MBTI·나이차 신뢰도 표기**: MBTI/궁합/나이차는 결정적 규칙 점수에 **반영하지 않는다**. 결과 화면([web/src/components/Report.tsx](../../web/src/components/Report.tsx))에서도 '성향·궁합 (참고)' 카드로 분리하고 "MBTI·나이차는 참고 요소, 핵심 판단은 신호 기준"임을 고지한다. 즉 **예측의 핵심 무기가 아니라 참여·개인화용 참고 레이어**다.

### 계층 2 — 결과 피드백 라벨 (자가발전의 핵심) ⭐ *(계획/미구현)*
**이게 없으면 자가발전 불가.** 진단 후 실제로 무슨 일이 있었는지 라벨링 *(현 MVP엔 결과 라벨 수집 기능 없음)*:
- 추천을 따랐나? (예/아니오/일부)
- 언제 행동했나?
- **결과: 긍정(따뜻한 반응·재회·고백 성공) / 중립 / 부정(거절·무응답·다툼)**

이 `(입력 → 추천 → 결과)` 삼중쌍이 쌓이면:
- 어떤 요인이 **실제로** 성공을 예측하는지로 **가중치 재보정**
- 손으로 튜닝한 규칙 → **학습 모델**(로지스틱/그래디언트 부스팅)로 P(성공 | 피처, 타이밍) 예측
- 추천 A/B 테스트로 검증

### 계층 3 — 전문가 라벨 (콜드스타트 해결) ⭐
결과 데이터가 쌓이기 전(초기엔 사례 0):
- **익명화된 실제 상담 사례 + 그 결과(outcome)** — 가장 가치 큼. ground truth라 가중치를 실제 성공률에 맞춰 보정 + **평가셋(eval)** 구축.
- **전문 상담사의 결정 규칙/케이스별 권고** — 심리 프레임워크(애착·재접촉 곡선)를 규칙으로 코드화.
- 좋은/나쁜 예시 메시지 코퍼스 — 메시지 생성 품질·톤 학습.

## 2. 자가발전 루프 (아키텍처) *(목표 아키텍처 — 계획/미구현)*
```
입력(설문+대화+자유서술)
   → 추천(점수·플랜·메시지)
   → 사용자 행동
   → 결과 라벨(긍정/중립/부정)  ← [플라이휠의 연료]
   → (주기적) 가중치 재보정 / 모델 재학습 + eval로 검증
   → 더 정확한 추천  ↺
```
- **마이그레이션 경로**: 규칙 엔진(**현재 구현**, 결정적·설명가능) → 규칙+학습 보정(하이브리드, 계획) → 학습 모델(데이터 충분 시, 계획). 점수의 결정성/설명가능성은 유지(신뢰 = 해자).

## 3. 데이터 스키마 추가 *(계획/미구현 — 현재는 `diagnoses` + `profiles` 테이블 존재, [web/supabase/schema.sql](../../web/supabase/schema.sql))*
현재 구현된 테이블은 `diagnoses`(진단 결과)와 `profiles`(생년·MBTI·애착 영속 입력, RLS 본인전용 select/insert/update)뿐이다. 아래 테이블/컬럼은 모두 **신규 추가 대상(자가발전 플라이휠용)**이다. 현재 `diagnoses`에는 `case_id` 컬럼도 없다.
```
outcomes      (신규) diagnosis_id, followed(bool), acted_at, outcome(pos/neu/neg), note, created_at
cases         (신규) 같은 관계를 묶는 단위(타임라인·추세 계산용)  ← diagnoses.case_id 컬럼도 신규 추가 필요
messages      (신규) 붙여넣은 대화(온도·응답속도 추출)  (민감 → 암호화/최소보관)
expert_cases  (신규, 오프라인 시드) 익명 사례 + 결과 라벨 → eval/보정용
```

## 4. 지금 founder가 넣어주면 좋은 것 (우선순위)
1. **익명화된 실제 상담 사례 N건 + 결과** (50~200건만 있어도 가중치 보정 + eval 하니스 제작 가능) ⭐
2. 전문가 결정 규칙/프레임워크 (콜드스타트 정확도)
3. 좋은/나쁜 연락 문구 예시 코퍼스 (메시지 품질)
→ 주면 내가: 가중치 보정 · eval 셋 · 프롬프트/규칙 개선까지 처리.

## 5. 주의 (함정)
- **생존편향**: 나쁜 결과는 보고를 안 함 → 후속 응답 인센티브·리마인더 필요.
- **표본 부족**: 초기엔 ML보다 전문가 규칙 + eval이 먼저. 섣부른 학습 금지.
- **프라이버시/동의**: 결과·대화는 민감정보 → 명시적 동의·암호화([backend](../backend/schema.md), 개인정보처리방침).
- **윤리**: '성공률'을 집착군에게 무기로 주지 않도록 위험군 분기 우선([at-risk-protocol](at-risk-protocol.md)).
