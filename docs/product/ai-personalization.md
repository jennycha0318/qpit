# AI 개인 상황별 커스터마이징 설계

> 개인의 구체적 상황에 맞춰 진단을 커스터마이징하는 방법.
> [ai-logic.md](ai-logic.md)의 LLM 적용편. 상태: 🚧 설계 v0.1 (계획 / 미구현)
>
> ⚠️ **현재 구현 상태(2026-06):** 이 문서가 설명하는 LLM 기반 AI 개인화는 **아직 구현되지 않았다.**
> 현재 web/ MVP의 진단 엔진은 **규칙기반·결정적**(`web/src/lib/diagnose/engine.ts`, `survey.ts`)이며 Claude/LLM API를 호출하지 않는다.
> 자유서술 입력란은 존재하지만 **진단 점수·문구에 반영되지 않고**, 기록으로만 저장된다(추후 개인화 재료). 아래 §2~§11은 향후 결합 설계다.

---

## 1. 핵심 원칙 — "규칙은 점수를, AI는 사람을"

> **점수·판단의 뼈대는 결정적 규칙이 잡고, AI는 그 위에 ‘그 사람의 언어’를 입힌다.**

| 역할 | 담당 | 이유 |
|------|------|------|
| **타이밍 점수·핵심 판단·가드레일** | 규칙 엔진 (frameworks) | 결정적·설명가능·일관성 → 신뢰 (PRD §11-2의 해자) |
| **근거 서술·맞춤 액션·연락 문구·톤** | LLM (Claude) | 개인 상황을 반영한 자연스러운 개인화 |

이렇게 분리하면:
- 같은 입력 → 같은 점수 (재현성·신뢰)
- LLM이 환각으로 "무조건 고백하세요" 같은 위험한 결론을 **점수로 뒤집을 수 없음** (안전)
- 그러면서도 결과 문장은 100% 그 사람 맞춤 (가치)

---

## 2. 아키텍처

```
[구조화 설문]  +  [자유서술 입력]  +  [상대 메시지(선택)]
        │
        ▼
 ① 규칙 엔진(frameworks)  →  timingScore · 핵심 요인 · 안전 플래그
        │
        ▼
 ② LLM 프롬프트 조립
    - system: 컨설턴트 역할 + 프레임워크 지침 + 가드레일
    - user: 정규화된 입력 + ①의 점수/요인/플래그 + 자유서술
        │
        ▼
 ③ Claude 호출 (구조화 JSON 출력 강제)
        │
        ▼
 ④ 검증 (zod 스키마 + 가드레일 후처리)
        │
        ├─ 성공 → 개인화 리포트
        └─ 실패/타임아웃 → ⑤ 규칙 기반 기본 문구로 폴백 (항상 결과 보장)
```

---

## 3. 개인화 입력 (3계층)

1. **구조화 설문** — 점수 계산용 핵심 변수 (현재 MVP: 상황별 8~9문항, `web/src/lib/diagnose/survey.ts`) — *구현됨*
2. **자유서술** — "무슨 일이 있었는지, 무엇이 고민인지" 자기 말로. AI 개인화의 핵심 재료 — *입력란은 구현됐으나, 현재는 점수/문구에 미반영하고 기록 저장만(추후 활용)*
3. **상대 메시지 붙여넣기 (선택)** — 받은 카톡을 넣으면 상대 톤·온도까지 반영 (v2 강력 기능) — *(계획 / 미구현)*

> 프라이버시: 자유서술/메시지는 민감정보. 익명 처리·최소 보관·삭제 정책 명시 (PRD §8)

---

## 4. AI가 커스터마이징하는 부분

| 출력 | 개인화 방식 |
|------|-------------|
| **진단 근거 서술** | 자유서술의 구체적 사실을 인용해 "당신의 경우…"로 풀어 설명 |
| **추천 액션** | 그 사람의 제약(거리·바쁨·상대 성향)에 맞게 액션을 구체화 |
| **연락 문구 초안** | 두 사람의 관계 톤·과거 대화 맥락을 반영한 실제 보낼 수 있는 문장 |
| **말투/톤** | 사용자의 애착유형·감정상태에 맞춰 위로형/직설형 조절 |
| **후속 Q&A** | "답장 안 오면?" 같은 추가 질문에 같은 맥락으로 이어서 답변 |

❗**개인화하지 않는 것:** 타이밍 점수, "지금 연락 금지" 같은 안전 판단 → 규칙이 고정.

---

## 5. 프롬프트 설계 (요지)

**System (고정·캐싱):**
- 역할: "근거 기반의 냉정하지만 따뜻한 연애 컨설턴트"
- 프레임워크 지침: 애착이론·재접촉곡선·상호성 (편향/아첨 금지)
- 가드레일: 집착·조작·스토킹 조장 금지, 비단정 프레이밍, 위기 시 중단+안내
- 출력 형식: 반드시 JSON 스키마(§6) 준수

**User (동적):**
```
[관계 단계] 이별 후 / [경과] 3주 / [이별사유] 환경 …
[규칙 엔진 결과] timingScore=72, label=now,
  핵심요인: +환경적 이유, +시간경과 / -상대 회피형
[자유서술] "마지막에 일 때문에 바빠서 소홀했고, 끝에 크게 싸웠어요…"
→ 위 점수·요인을 존중하되, 자유서술을 반영해 개인화 리포트를 JSON으로 생성하라.
```

---

## 6. 출력 스키마 (zod)

```ts
{
  reason: string,            // 개인화된 진단 근거 (자유서술 인용)
  actions: { title, detail }[],   // 1~3개 맞춤 액션
  risks: string[],           // 주의·가드레일
  message: string | null,    // 연락 문구 (보내면 안 될 땐 null)
  holdNote: string | null,   // 보내지 말아야 할 때의 안내
  tone: "warm" | "direct"    // 적용된 말투
}
// timingScore/label은 규칙 엔진 값을 그대로 사용 (LLM이 생성하지 않음)
```

## 7. 안전·품질 장치

- **점수 잠금** — LLM 출력에 점수를 맡기지 않음 → 위험 결론 원천 차단
- **스키마 검증** — 형식 불일치 시 재시도, 반복 실패 시 폴백
- **가드레일 후처리** — 금지 표현·집착 권유 감지 시 차단
- **위기 감지** — 자해·폭력·학대 키워드 → 컨설팅 중단 + 전문기관 안내
- **항상 결과 보장** — LLM 실패해도 규칙 기반 기본 리포트 제공

---

## 8. 비용·지연 (UX)

- **모델 선택:** 기본 진단은 빠른 모델, 심층/유료는 상위 모델 (claude-opus-4-8)
- **프롬프트 캐싱:** system 프롬프트 캐싱으로 비용·지연 절감
- **스트리밍:** 리포트를 토큰 단위로 흘려 체감 대기 단축
- **무료/유료 분기:** 무료는 규칙+짧은 개인화, 유료는 풀 개인화+메시지 생성

---

## 9. 데이터 플라이휠 *(계획 / 미구현)*

진단 → 사용자가 결과 채택 여부·실제 결과 피드백(Feedback 모델) →
잘 맞은/틀린 케이스를 모아 **프롬프트·규칙 가중치 개선**. 시간이 갈수록 개인화 정확도↑ (PRD §11-2 해자).

> 현재 MVP에는 결과 피드백 수집/Feedback 테이블이 없다. DB는 `diagnoses` 테이블만 운영 중(`web/supabase/schema.sql`).

## 10. 현재 구현 상태 / 데모 계획

**현재 web/ MVP (2026-06):**
- 설문 마지막에 **자유서술 입력란**이 있다(`web/src/lib/diagnose/survey.ts`의 `freeText`, `web/src/app/diagnose/page.tsx`).
- 단, 이 자유서술은 **진단 점수·문구·요인 어디에도 반영되지 않는다.** 로그인 시 결과(diagnoses)에 함께 저장만 되고, "추후 AI 개인화에 활용될 예정"으로 명시돼 있다.
- 점수·요인·플랜·메시지는 모두 규칙 엔진(`engine.ts`)이 결정적으로 산출한다 (원칙 1).

**(계획 / 미구현) — 아직 없는 데모 요소:**
- 입력 텍스트에서 테마(다툼·바쁨·그리움·새사람 등)를 감지하는 mock 개인화
- "✨ AI가 추가로 고려한 점" 카드 및 텍스트 기반 맞춤 팁·문구
- 브라우저에서 실제 Claude를 호출할 수 없으므로(키 노출), 데모 단계에서는 위 mock으로 UX를 시연할 수 있다.

→ 실제 제품에서는 이 자리에 §5의 Claude 호출(서버 API)이 들어간다.

## 11. 실제 구현 로드맵 (web/)

현재 규칙 엔진은 `web/src/lib/diagnose/`(`survey.ts`·`engine.ts`)에 **구현돼 있다.**
아래 경로는 **계획 / 미구현**이다 — `web/src/lib/frameworks/`와 `web/src/lib/ai/`에는 현재 설계 README만 있고(`.ts` 구현 없음), `web/src/app/api/`도 비어 있다(`.gitkeep`).

```
[구현됨] web/src/lib/diagnose/
  ├─ survey.ts            # 상황별 설문 정의
  └─ engine.ts            # 규칙기반·결정적 진단 엔진

[계획/미구현] web/src/lib/frameworks/   # 규칙 레이어 형식화 (현재 README 뿐)
[계획/미구현] web/src/lib/ai/
  ├─ prompts/diagnose.ts  # system/user 프롬프트
  ├─ schema.ts            # zod 출력 스키마 (§6)
  └─ diagnose.ts          # 규칙→LLM→검증→폴백 오케스트레이션
[계획/미구현] web/src/app/api/diagnose/route.ts   # 서버 API (키 보호)
```
