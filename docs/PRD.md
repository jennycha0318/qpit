# Pacemaker — 제품 기획서 (PRD)

> AI 기반 연애·재회 의사결정 컨설팅 서비스
> 버전 0.1 · 작성일 2026-06-15

---

## 1. 한 줄 요약 (Elevator Pitch)

> **"고백해도 될까? 지금 연락해도 될까?"**
> 연애의 결정적 순간마다, 내 상황과 상대 성향 데이터를 분석해
> **언제·어떻게 행동해야 성공 확률이 높은지** 알려주는 AI 연애 컨설턴트.

---

## 2. 문제 정의 (Problem)

연애·재회에서 사람들이 가장 고통받는 지점은 **"감정"이 아니라 "의사결정"**이다.

- **타이밍의 불확실성** — "지금 고백하면 부담스러울까? 더 기다려야 할까?"
- **재회의 막막함** — "헤어진 지 3주, 지금 연락하면 매달리는 걸로 보일까?"
- **객관적 시각의 부재** — 친구는 내 편만 들고, 나는 감정에 매몰되어 판단이 흐려진다.
- **행동의 결과 예측 불가** — 한 번의 잘못된 메시지가 관계를 끝낼 수 있다는 두려움.

기존 해결책의 한계:
| 대안 | 한계 |
|------|------|
| 친구 상담 | 주관적, 편향됨, 매번 부탁하기 미안함 |
| 연애 유튜브/책 | 일반론, 내 구체적 상황에 안 맞음 |
| 연애 상담 (유료) | 비쌈, 즉시성 없음, 접근성 낮음 |
| 사주/타로 | 근거 없음, 일회성 |

→ **개인화 + 즉시성 + 데이터 근거**를 동시에 충족하는 솔루션이 없다.

## 3. 타겟 사용자 (Target Users)

### 핵심 페르소나

**① "썸 정체기" 지수 (28세, 직장인)**
- 6개월째 호감 가는 동료와 썸 중. 고백 타이밍을 놓칠까 불안.
- 니즈: "언제 고백해야 거절당하지 않을까?"

**② "재회 희망" 민준 (31세, 직장인)**
- 3년 연애 후 한 달 전 이별. 다시 잘해보고 싶지만 어떻게 시작할지 모름.
- 니즈: "지금 연락해도 될까? 뭐라고 보내야 할까?"

**③ "관계 불안" 서연 (25세, 대학원생)**
- 연애 중이지만 상대 마음이 식은 것 같아 불안. 행동 하나하나 고민.
- 니즈: "이 상황에서 내가 어떻게 해야 관계가 유지될까?"

### 시장 규모 (가설 — 추후 검증)
- 1차: 국내 20–35세 연애 관심층
- 진입 웨지: **재회 니즈** (절박함 = 지불 의향 높음, 검색량 높음)

## 4. 핵심 가치 제안 (Value Proposition)

1. **개인화된 의사결정 지원** — 일반론이 아닌 "내 상황"에 대한 구체적 답.
2. **타이밍 추천** — "언제" 행동해야 하는지에 대한 명확한 신호.
3. **액션 가이드** — "무엇을, 어떻게" (연락 문구, 만남 제안 방식 등) 구체적 제안.
4. **확률적 프레이밍** — "성공/실패" 단정이 아닌 "지금 행동 시 vs 기다릴 시" 비교.
5. **즉시성 + 익명성** — 새벽 3시에도, 누구에게도 말 못 할 고민을 털어놓을 수 있음.

## 5. 핵심 기능 (Core Features)

### MVP (v1, 현재 구현됨) — "재회/고백 타이밍 진단"
1. **상황 입력 (온보딩 설문)** ✅
   - 나이대 선택 (10대 / 20대 / 30대 / 40대 이상) → 10대는 청소년 눈높이 모드
   - 관계 단계 (썸 / 연애 중 / 이별 후)
   - 핵심 사실 (만난 기간, 헤어진 시점, 이별 사유, 연락 추세 등 — 단계별 설문)
   - 내 성향 & 상대 성향 (애착유형 간이 진단)
   - 비로그인(게스트)도 진단 가능, 로그인 시 결과를 히스토리에 저장
2. **진단 리포트** ✅ (규칙 기반 — 아래 6번 참조, 현재 LLM 미사용)
   - 현재 상태 분석 (객관적 거울) — 점수 + 요인(factors)
   - **타이밍 신호** — "지금 / N일 후 / 더 기다림" + 근거 (when/channel/단계별 plan)
   - **액션 추천** — 구체적 행동 1–3개
   - **하지 말아야 할 것** — 흔한 실수·리스크 경고
3. **추천 메시지 예시** ✅ — 상황 맞춤 연락 문구 초안 제공 (현재는 점수 구간별 정적 템플릿 / LLM 생성은 계획)

### v2 이후 (확장 — 미구현/계획)
- **시뮬레이션** — "이렇게 보내면 상대가 어떻게 반응할까?" 응답 예측 (계획)
- **타임라인 트래킹** — 진행 상황 기록하며 단계별 컨설팅 (계획)

- **LLM 기반 개인화** — 자유서술 입력을 반영한 맞춤 진단/문구 생성 (계획)

- **집착 위험군 쿨다운** — 반복 진단·과도 연락 신호 시 진단 제한 (계획)

## 6. AI 추천 로직 설계 (핵심 차별점)

> ⚠️ 정직성 원칙: 우리는 미래를 "예언"하지 않는다.
> **심리학 프레임워크 + 입력 데이터**를 결합해 "확률을 높이는 합리적 선택"을 제안한다.

### 입력 (Inputs)
- **상황 변수**: 관계 단계, 경과 시간, 연락 빈도/추세, 마지막 상호작용 톤
- **성향 변수**: 애착 유형(안정/불안/회피), 연애 가치관, 갈등 대응 방식
- **맥락 변수**: 이별 사유, 미해결 갈등 유무, 외부 요인(바쁨/이사 등)

### 분석 프레임 (Reasoning Frameworks)
- 애착 이론 (Attachment Theory) — 상대 유형별 적정 접근 속도
- No-contact / 재접촉 곡선 — 이별 후 시간 경과와 재회 수용도
- 상호성·희소성 원칙 — 과도한 연락 vs 적정 거리
- 손실 회피 — "지금 행동 안 함"의 기회비용 제시

### 출력 (Outputs)
- **타이밍 점수** (예: "지금 연락 적정도 72/100") + 한 줄 근거 — ※현재 엔진은 3~97로 clamp
- **권장 액션** (우선순위순) + 이유
- **리스크 경고** (이 상황에서 흔한 실수)
- **대안 시나리오** (A안 vs B안 비교 — 현재는 점수 구간별 단일 플랜 제시, 분기 비교는 계획)

### 구현 방식 (기술 — 현재 MVP)
- **현재: 규칙 기반(결정적) 엔진** — LLM/AI API 호출 없이 `web/src/lib/diagnose/survey.ts`(설문)·`engine.ts`(점수·요인·플랜·메시지)로 산출. 같은 입력은 항상 같은 결과 → 신뢰·일관성 확보.
- 흐름: 설문 입력 → 요인별 가중치 합산 → 점수 clamp(3~97) → 구간별 액션·리스크·플랜·메시지 템플릿 렌더링.
- 안전 플래그를 엔진에서 직접 계산: `needsSupport`(정서 위기 신호), `minor`(10대) — 아래 8번 연동.
- **계획: LLM(Claude API) 결합** — 자유서술 입력 정규화 → 프레임워크 프롬프트 → 구조화 출력(JSON)으로 개인화 강화 (현재 미구현).

## 7. 데이터 모델

### 현재 구현 (Supabase Postgres) — `web/supabase/schema.sql`

```
diagnoses  (진단 결과 1건 = 한 행)
 ├─ id            uuid  (PK)
 ├─ user_id       uuid  (default auth.uid(), → auth.users, on delete cascade)
 ├─ stage         text  (crush | dating | breakup)
 ├─ score         int
 ├─ result        jsonb (Diagnosis 전체: factors/actions/risks/plan/msg/needsSupport/minor 등)
 └─ created_at    timestamptz
   * RLS: 본인 행만 select/insert/delete (update 정책은 미정의)
   * 인증: Supabase Auth (이메일·비번 + Google OAuth + 비번 재설정)
   * 비로그인(게스트) 진단은 저장하지 않음
```

### 확장 모델 (계획 — 미구현)

```
User / Profile  (나이대·성별·애착유형·연애가치관) — 현재 별도 테이블 없음(나이대는 진단 시 클라이언트 입력)

Case (상담 케이스 — 한 건의 고민)
 ├─ id, userId, 관계단계(썸/연애/이별후)
 ├─ 핵심사실: 만난기간, 마지막연락일, 이별사유, 연락추세
 ├─ 상대성향: 애착유형(추정), 특징 태그
 └─ createdAt

Diagnosis (진단 결과) — caseId 연결, generatedMessages[] 등
Event — 케이스 타임라인 이벤트 (연락함/만남/반응 등)
Outcome / 결과 피드백 — 데이터 플라이휠용
```
> 현재 MVP는 단일 `diagnoses` 테이블만 사용한다. 위 Case/Event/Outcome 등 확장 테이블은 계획 단계다.

## 8. 윤리·안전 가드레일 (필수)

연애 컨설팅 서비스는 오·남용 시 위험이 크다. MVP부터 내장한다.

**현재 구현됨 ✅**
- **집착·스토킹 방지** — 차단·거절·무응답·과도 연락 신호 시 "더 시도" 대신 "물러서기" 권고, 우회 연락 금지 명시. (단, 반복 진단 자체를 막는 '쿨다운'은 미구현)
- **건강한 관계 우선** — 조작(가스라이팅·밀당 악용)이 아닌 진정성 기반 행동 권장.
- **비단정·확률 프레이밍** — "무조건 성공" 같은 표현 금지, 적정도 점수로 제시.
- **위기 대응** — 위기 신호(`needsSupport`) 시 상담 연결 카드 노출: 자살예방 상담(전화 109, 24시간), 청소년 상담(9~24세 지역번호+1388 / 문자 #1388). (컴포넌트: `CrisisResources`)
- **법적·윤리 고지** — 외도·학대 등은 법적·윤리 문제가 될 수 있음을 사전 고지하고 대한법률구조공단 무료 법률상담 링크 제공. (`LegalEthicsNotice`)
- **청소년(10대) 모드** — 미성년 선택 시 눈높이 톤 + 지지 배너(`MinorSupportBanner`).
- **프라이버시** — 개인정보 동의 + `/privacy` 처리방침, 비로그인(익명) 사용 가능, 게스트 진단은 저장 안 함.

**계획 (미구현)**
- 입력 텍스트 기반 자동 위기 정황 탐지(자해/폭력/학대 NLP) — 현재 위기 플래그는 설문 응답 규칙으로만 산출.
- 집착 위험군 진단 쿨다운(재진단·연락 빈도 제한).

## 9. 사용자 플로우 (현재 MVP)

```
랜딩(/) → 나이대 선택 → 고민 유형 선택(썸/연애/이별후) → 상황·성향 설문(2~3분)
     → [규칙 기반 진단] → 진단 리포트(점수·플랜·액션·리스크·메시지 예시)
     → (CTA) 다시 진단 / 로그인 시 히스토리 저장
```
- 진단(`/diagnose`)은 로그인 없이도 가능, 로그인 사용자는 결과가 `diagnoses`에 저장되고 `/history`에서 조회.
- 주요 라우트: `/`(랜딩), `/login`, `/signup`, `/reset-password`, `/update-password`, `/auth/callback`, `/diagnose`, `/history`, `/history/[id]`, `/profile`, `/privacy`. (`/home`은 `/diagnose`로 리다이렉트)
- '공유(share)'는 현재 미구현(계획). 분석은 LLM이 아닌 규칙 기반(결정적).

### 11-2. 범용 LLM(ChatGPT·Claude·Gemini) 대비 차별점

> 가장 강력한 경쟁자는 다른 연애앱이 아니라 **"그냥 ChatGPT에 물어보면 되잖아?"** 다.
> 이 질문에 답하지 못하면 사업이 성립하지 않는다. 정직하게: **모델 자체로는 못 이긴다.
> 우리의 해자(moat)는 모델 위에 쌓는 "도메인 시스템"이다.**

| 비교축 | 범용 LLM (ChatGPT 등) | Pacemaker |
|--------|----------------------|-----------|
| **입력 품질** | 사용자가 뭘 말해야 할지 모름 → 빈약한 정보로 빈약한 답 (garbage in, garbage out) | 의사결정에 필요한 변수(애착유형·연락추세·경과시간·이별사유)를 **구조화 설문으로 강제 수집** |
| **답변 성향** | 듣고 싶은 말을 해주는 경향(sycophancy) → "용기 내서 고백하세요!" 식 응원 | 의도적 **"냉정한 거울"** + 하지 말아야 할 것 경고. 편향 보정이 설계에 내장 |
| **출력 형태** | "상황에 따라 달라요" 두루뭉술한 산문 | **타이밍 점수·우선순위 액션·리스크** 등 의사결정 가능한 구조화 출력 |
| **전문성** | 그때그때 일반론, 일관성 없음 | 검증된 심리 프레임(애착이론·재접촉곡선)을 **규칙 레이어로 강제 적용** → 일관된 전문가 관점 |
| **연속성** | 세션마다 처음부터, 내 케이스 기억 못 함 | 케이스를 **기억하고 시간축으로 추적** — 연애는 한 번의 답이 아니라 "페이스 메이킹" 문제 |
| **안전성** | 집착·조작(밀당 악용·가스라이팅)도 요청하면 거들어줌 | 안티 스토킹·건강한 관계 **가드레일 내장**, 위기 신호 시 개입 |
| **행동 전환** | "잘 얘기해보세요" 추상적 | 즉시 쓸 수 있는 **연락 문구 초안 + 액션 체크리스트** 산출 |
| **접근성** | 빈 채팅창 공포, 프롬프트 잘 써야 좋은 답 | 2–3분 설문이면 끝, **프롬프트 역량 불필요** |
| **개선 루프** | 이 도메인의 결과 피드백 없음 | 실제 결과("연락했더니 이렇게 됨") 수집 → **추천 품질 데이터 플라이휠** |

#### 한 문장 요약
> ChatGPT는 **"무엇을 물어야 할지 아는 사람"**에게만 좋은 답을 준다.
> Pacemaker는 **무엇을 물어야 할지, 무엇을 답해야 할지, 언제 다시 와야 할지**를 시스템이 안다.
> = 범용 LLM은 **재료**, Pacemaker는 **완성된 의사결정 시스템**.

#### "그냥 래퍼 아니냐"에 대한 답 (해자의 본질)
1. **구조화된 데이터 수집 자산** — 어떤 질문을 해야 좋은 진단이 나오는지에 대한 노하우
2. **검증된 프레임워크 + 규칙 레이어** — sycophancy를 억제하고 전문가 판단을 재현
3. **타임라인 메모리** — 한 번의 답이 아닌 관계의 시간축 전체를 컨설팅
4. **결과 피드백 데이터 플라이휠** — 시간이 갈수록 추천이 정교해지는 선순환
5. **신뢰·전용성 브랜드** — 민감한 고민을 털어놓을 수 있는 맥락 전용 안전지대

→ 모델이 좋아질수록 우리도 같이 좋아진다(모델 = 엔진, 우리 = 차). 모델 발전은 위협이 아니라 순풍.

## 12. 리스크 & 검증 과제

- **정확성/신뢰 리스크** — 추천이 빗나가면 신뢰 붕괴. → 확률·근거 투명 공개로 완화.
- **윤리 리스크** — 집착 조장 논란. → 8번 가드레일.
- **수요 검증** — 진짜 돈을 낼까? → 랜딩+사전예약으로 검증.
- **리텐션** — 일회성 소비 우려. → 타임라인 컨설팅으로 지속 사용 유도.
